# [\#14751 Issue](https://github.com/ethereum/solidity/issues/14751) `closed`: Invalid string values in smtchecker counterexample for a failed assertion
**Labels**: `bug :bug:`, `smt`


#### <img src="https://avatars.githubusercontent.com/u/23142088?u=5d4bf7b0dd787e74d3a26cda1cb2d0f5c109da20&v=4" width="50">[pgebal](https://github.com/pgebal) opened issue at [2023-12-22 16:02](https://github.com/ethereum/solidity/issues/14751):

In `smtCheckerTests/types/string_1.sol` the counterexample is invalid.

```
contract C
{
	function f(string memory s1, string memory s2) public pure {
		assert(bytes(s1).length == bytes(s2).length);
	}
}
```
running the smtchecker produces the following warning:
```
Warning 6328: (77-121): CHC: Assertion violation happens here.\nCounterexample:\n\n\nTransaction trace:\nC.constructor()\nC.f(s1, s2)
```

There are no literal string values in the counterexample.
Even when `s1` and `s2` were interpreted as literals, the values do not fail the assertion.

#### <img src="https://avatars.githubusercontent.com/u/23142088?u=5d4bf7b0dd787e74d3a26cda1cb2d0f5c109da20&v=4" width="50">[pgebal](https://github.com/pgebal) commented at [2024-02-05 15:14](https://github.com/ethereum/solidity/issues/14751#issuecomment-1927229183):

The reason is that values generated by z3 are longer than 1024 characters.
We can either check if it's possible to force solver to produce shorter strings or to print a warning that paramter name is used instead of a concrete value.
